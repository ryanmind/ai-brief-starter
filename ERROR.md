# ERROR 记录（2026-03-04）

用于记录当前仓库中已确认、但尚未彻底修复的关键问题，便于后续跟进。  
文档站镜像页：`docs/error.md`。

## 1) 通知失败被静默吞掉（高优先级）

- 位置：`.github/workflows/daily.yml`
- 现象：`Notify Feishu`、`Notify WeChat` 使用 `continue-on-error: true`，通知失败不阻断流程，容易误以为一切正常。
- 影响：日报已生成但无人收到飞书/微信提醒，值守与运营感知滞后。
- 建议：
  - 至少在通知失败时追加显式告警（如单独 failure 通知或 summary）。
  - 区分“可选通知失败”和“配置错误导致必然失败”。

## 2) AI 主题过滤误判（高优先级）

- 位置：`src/config.py`、`main.py`
- 现象：当前基于简单子串匹配（如关键词 `ai`），容易把非 AI 内容误判为 AI。
- 影响：早报混入非 AI 条目，降低内容质量与可信度。
- 建议：
  - 将关键词匹配升级为词边界/正则策略。
  - 对可疑条目增加二次判定（建议 LLM 判别 + 置信阈值）。

## 3) Docs 部署触发重复与竞态（中高优先级）

- 位置：`.github/workflows/docs.yml`
- 现象：同时监听 `push` 与 `workflow_run`，且并发分组键场景不完全一致，可能重复部署或出现非最新版本。
- 影响：站点偶发延迟、重复构建、部署结果不可预期。
- 建议：
  - 收敛触发路径（优先保留单一主触发）。
  - 统一 concurrency key 的分支解析策略。

## 4) README 与实际权限不一致（已修复）

- 位置：`README.md`
- 现象（历史）：文档曾写 `contents: read` 即可，但 daily workflow 需要 `contents: write`（会回写 docs）。
- 影响：新维护者按旧文档配置，容易踩权限坑。
- 当前：已在 `README.md` 完成权限说明对齐。

## 5) 质检策略需改造（中优先级）

- 位置：`.github/workflows/daily.yml`
- 现象：`QUALITY_CHECK_FAIL_OPEN=1`，质量缺陷不会阻断发布。
- 影响：在“必须持续出刊”的前提下，低质量标题/摘要可能直接上线。
- 业务约束：**必须“有内容产出”**，不允许因为质检失败而断更。
- 建议（兼顾不断流与质量）：
  - 保持 fail-open，不做全局 fail-close。
  - 对关键缺陷做自动兜底（标题补全、摘要抽取式回退、非 AI 条目剔除）。
  - 输出“降级发布/待复核”标记，并同步飞书/微信告警。
