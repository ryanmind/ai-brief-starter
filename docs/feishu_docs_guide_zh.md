# 飞书文档使用详细教程（从 0 到可协作）

适用对象：第一次使用飞书文档，或想把团队信息沉淀到飞书的人。  
目标：你能独立完成“创建文档 → 协作编辑 → 权限管理 → 发布分享 → 版本回溯”。

---

## 0. 飞书入口地址（先收藏）

- [飞书官网](https://www.feishu.cn/)
- [飞书网页版入口](https://feishu.cn/)
- [飞书开放平台（做自动化/机器人时用）](https://open.feishu.cn/)

---

## 1. 先理解：飞书文档能解决什么问题

- 统一资料：把会议纪要、方案、周报放在同一套空间里。
- 多人协作：多人同时编辑，实时看到谁改了什么。
- 可追溯：保留版本历史，误删可恢复。
- 可分享：按人、按群、按链接控制可见范围。

---

## 2. 准备工作（5 分钟）

1. 注册/登录飞书账号（个人或企业均可）。
2. 安装桌面端或使用网页版。
3. 确认你已加入团队（企业场景建议）。

目的：确保后续能正常创建文档与分配权限。

---

## 3. 第一步：创建文档

1. 打开飞书工作台，进入「文档」。
2. 点击「新建」→ 选择「文档」。
3. 输入标题，例如：`AI日报-项目知识库`。
4. 在文档开头写“文档目的、更新时间、负责人”。

建议模板：

```text
文档目的：
负责人：
最近更新：
目录：
```

作用：后续协作时，任何人都能快速看懂文档定位。

---

## 4. 第二步：搭建文档结构（非常关键）

建议按 4 层结构：

1. 概览（目标、范围、状态）
2. 过程（步骤、记录、问题）
3. 结果（结论、产出、链接）
4. 附录（术语、参考、变更记录）

操作技巧：

- 使用一级/二级标题形成目录。
- 每段不超过 5-8 行，便于阅读。
- 关键结论用加粗，避免埋在长段落里。

---

## 5. 第三步：多人协作编辑

1. 点击右上角「分享」。
2. 添加协作者（同事邮箱/用户名/群组）。
3. 分配权限：
   - 可编辑：参与撰写
   - 可评论：只能提意见
   - 可查看：只读
4. 打开评论模式，对具体句子提建议。

作用：把“口头沟通”变成“结构化协作”，减少信息丢失。

---

## 6. 第四步：权限与安全设置

建议按文档敏感度分级：

- 普通资料：团队可查看，核心成员可编辑
- 业务敏感：仅指定成员可访问
- 外部共享：仅开启“可查看”，关闭编辑权限

实操建议：

1. 不要长期使用“任何人可编辑”。
2. 对外分享前，先检查是否含内部信息。
3. 定期（每月）复查分享名单与链接权限。

---

## 7. 第五步：版本历史与恢复

当你遇到误删/改坏内容时：

1. 打开文档右上角菜单。
2. 进入「历史版本」。
3. 找到目标时间点的版本。
4. 预览后选择恢复。

作用：避免“最后一次保存覆盖全部”带来的损失。

---

## 8. 第六步：发布与通知团队

推荐发布流程：

1. 文档顶部标注状态：`草稿 / 待评审 / 已发布`。
2. 发布时在群里发送：更新摘要 + 文档链接 + 负责人。
3. 明确反馈截止时间（例如：今晚 18:00）。

作用：减少“发了没人看”的情况，提高执行效率。

---

## 9. 常见高效用法

- 会议纪要：固定模板（结论、待办、负责人、截止时间）。
- 项目周报：本周进展 / 风险 / 下周计划。
- SOP 文档：步骤编号 + 截图 + 常见报错。
- 决策记录：背景、选项、结论、影响范围。

---

## 10. 常见问题与处理

### Q1：别人看不到文档

- 检查是否已添加对方；
- 检查对方是否在正确租户；
- 检查链接权限是否过严。

### Q2：误删后找不回

- 先看历史版本；
- 再看回收站；
- 仍无法恢复时联系管理员。

### Q3：文档越来越乱

- 固定命名规范：`项目-主题-日期`；
- 每周整理一次目录；
- 把过期内容归档到“历史”章节。

## 11. 给你当前项目的落地建议（ai-brief-starter）

你可以把本项目输出的 `reports/latest.md` 同步到飞书文档，建议流程：

1. 在飞书建一个文档：`AI早报总览`。
2. 每天新增一个二级标题：`YYYY-MM-DD`。
3. 将 `reports/latest.md` 内容粘贴进当天章节。
4. 文末保留来源链接，保证可追溯。
5. 在飞书群里发“今日更新摘要 + 文档链接”。

这样做的作用：

- 对内：沉淀可检索的日报库；
- 对外：可基于文档继续二次加工（周报、复盘、专题）。

---

## 12. 建议的执行节奏（可直接照做）

- 每天 1 次：更新日报文档 + 群通知。
- 每周 1 次：整理目录与归档过期内容。
- 每月 1 次：权限复查 + 模板优化。

按这个节奏执行，文档会长期可用，而不是“写一次就废弃”。

## 13. 本仓库定时任务同步到飞书（一步一步）

下面是 `ai-brief-starter` 的实操流程。

### Step 1：在飞书群创建机器人

1. 进入目标群聊。
2. 点击「群设置」→「群机器人」→「添加机器人」。
3. 选择「自定义机器人」。
4. 复制 Webhook 地址。
5. 如果你开启了“签名校验”，同时记录 Secret。
6. 如果开启了“关键词校验”，请至少添加关键词：`AI早报`。

作用：让 GitHub Actions 能把消息推送到飞书群。

### Step 2：在 GitHub 配置 Secrets

路径：`Settings -> Secrets and variables -> Actions`

新增以下密钥：

- `FEISHU_WEBHOOK_URL`：机器人 Webhook（必填）
- `FEISHU_BOT_SECRET`：签名 Secret（仅当你在飞书开启签名时填写）

作用：避免把凭证写死在代码里。

### Step 3：确认仓库脚本与工作流

本仓库已包含：

- 脚本：`scripts/notify_feishu.py`
- 工作流步骤：`.github/workflows/daily.yml` 里的 `Notify Feishu`

执行顺序是：

1. 生成日报 `reports/latest.md`
2. 质量检查
3. 回写报告到仓库
4. 推送飞书通知

### Step 4：手动触发一次验证

先确认：你已经把本地改动 `commit + push` 到远程分支（否则 Actions 仍运行旧代码）。

1. 打开 GitHub Actions。
2. 选择 `ai-morning-brief`。
3. 点击 `Run workflow`。
4. 观察 `Notify Feishu` 步骤日志应为 `notify success`。
5. 飞书群里应收到“标题 + 今日要点 + 任务详情链接”。

### Step 5：常见报错排查

- 没消息但步骤成功：检查群里是否有关键词限制。
- `HTTP 403/401`：Webhook 或签名错误。
- 步骤被跳过：`FEISHU_WEBHOOK_URL` 为空。
- 文案为空：确认 `reports/latest.md` 有「今日要点」段落。
