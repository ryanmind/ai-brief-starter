# GH Scheduled

## 何时使用

- 任务涉及仓库中的定时任务或自动化流程。
- 你要修改调度时间、触发条件、输入源、source 配置、处理逻辑、输出产物、通知、质检或相关 CI / GitHub Actions。

## 核心规则

- 先区分“源数据 / 处理逻辑 / 输出产物 / 调度配置”，不要混着改。
- 优先修改源代码、配置和工作流，不要直接手改生成产物。
- 先做最小范围修复，再决定是否扩展到上下游。
- 行为变化要补聚焦的回归测试；没有测试体系时，不额外引入重型测试框架。
- 先跑最小验证命令，通过后再扩大验证范围。

## 标准流程

1. 先明确任务属于哪一层：调度、采集、处理、生成、同步、通知或质检。
2. 只阅读与该层直接相关的文件和配置。
3. 在真正的源头位置修复问题，不在生成结果或临时产物上打补丁。
4. 如涉及时间、cron 或“今天/明天”等相对日期，统一检查时区与绝对日期。
5. 完成后按从小到大的顺序验证：单测、局部脚本、完整流程。

## 文件检查清单

- 调度配置：`.github/workflows/`、crontab、任务入口脚本
- 输入与采集：信源清单、API 客户端、抓取脚本、环境变量
- 处理逻辑：主流程入口、过滤器、清洗、聚合、排序、模板
- 输出与同步：报告目录、发布脚本、文档同步、通知脚本
- 质量保障：测试目录、质量检查脚本、健康检查脚本

## Source 配置

- 优先把输入源做成可配置项，放在配置文件、环境变量或 workflow inputs 中，不要写死在代码里。
- 明确 source 的关键参数：鉴权信息、抓取时间窗口、超时、重试、去重键、默认值。
- 修改 source 时，同时检查上游输入格式和下游处理逻辑是否仍兼容。
- 若 source 依赖外部服务，说明本地验证前置条件，并补充失败回退或降级策略。

## 复杂任务拆分

- 采集层：负责 source 配置、鉴权、抓取窗口、重试和输入格式确认。
- 处理层：负责清洗、过滤、聚合、排序、模板与输出一致性。
- 交付层：负责 workflow、同步、通知、发布与验证收口。

## 验证原则

- 先运行最小相关命令，再运行完整任务。
- 优先验证时间窗口、幂等性、重复执行、空数据和失败重试。
- 若任务依赖密钥或外部服务，明确说明前置条件与无法在本地完成的部分。
